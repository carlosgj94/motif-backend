# Content Parser Real-Site Intake

Date: 2026-04-05

This document records the first batch of real URLs proposed for parser
validation. It is intentionally operational: which branch each URL should map
to, whether a plain HTML fetch works today, and what the current parser returns
for the pages we can fetch openly.

These URLs came from product-driven examples rather than synthetic test cases.
They should become captured fixtures over time.

## Current Parser Observations

These observations were generated by fetching the openly accessible pages with
`curl -L`, replaying them through `parseFetchedDocument(...)`, and disabling
favicon fetches to keep the run deterministic.

## Blocked Capture Notes

### X URLs

Both X URLs return `403` to plain `curl -L -I` from this environment. That is
not a parser failure by itself; it is a capture-path constraint. The current
capture strategy for real fixtures uses the official public X endpoints instead:

- `publish.twitter.com/oembed`
- `cdn.syndication.twimg.com/tweet-result`

These URLs still matter because they exercise:

- long-post extraction
- X article/post format distinctions
- metadata fallback versus markup extraction

Current X coverage in the shared fixture corpus:

- `x/karpathy_note_tweet_real_capture` covers a real note-tweet capture through
  official public sidecars and guards against falsely promoting a teaser into
  an article.
- `x/kevingu_article_real_capture` covers a real X Article share and proves the
  parser can upgrade a link-only post into article-style output using the
  official `article` object.
- `x/single_post` covers visible-markup post extraction plus avatar filtering.
- `x/long_post` covers payload-script enrichment and long-form single-post
  normalization into article-style blocks using a synthetic seed fixture where
  the full long-form payload is present.

### Bloomberg URL

The Bloomberg feature URL returns `403` to plain `curl -L -I`. This is an
important case because Bloomberg is exactly the sort of high-value source where
the parser must either:

- succeed on the public article HTML when accessible
- use a lawful fallback such as an archived snapshot
- or later route to a rendered or alternative capture path if we add one

Current synthetic coverage in the shared fixture corpus:

- `bloomberg/ai_perfected_chess` covers the direct `bloomberg-article` adapter
  with Bloomberg-specific promo cleanup and metadata recovery.
- `archive/bloomberg_snapshot` covers archive-host delegation into the same
  Bloomberg extraction heuristics.
- `archive/xataka_snapshot` covers the generic archive delegation path and
  protects against archive boilerplate, related-content rails, and div-wrapped
  article bodies regressing generic extraction.

### `archive.ph/yn2It`

The archive URL returns `429` to plain `curl -L -I`, which matches why the
archive mirror fallback logic already exists in the current content processor.

This URL should become a real archive fixture once we have a stable way to
capture one of the archive mirrors without immediately tripping rate limits.

Current archive status:

- `archive.ph`, `archive.is`, `archive.md`, `archive.vn`, and `archive.li`
  returned `429` from this environment during capture attempts.
- Archive handling is still moving forward through source-aware fixture
  coverage, but `archive/archive_yn2it` remains blocked on a reliable real
  capture path.

synthetic seeds to real pages. This list is the first practical bridge between
the harness and the real parser backlog.
